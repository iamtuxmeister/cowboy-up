# cowboy-up

A project generator for [Erlang](https://www.erlang.org/) web applications using [Cowboy](https://ninenines.eu/docs/en/cowboy/2.10/guide/), [ErlyDTL](https://github.com/erlydtl/erlydtl), and SQLite. Inspired by `create-react-app` and Rails generators — one command to scaffold a production-ready project, another to generate models with relationships.

```
cowboy-up new "bible site" --css pico
```

---

## Features

- **Project scaffolding** — full Cowboy/ErlyDTL/SQLite project in one command, with interactive CSS, template engine, and database choices
- **CSS frameworks** — Basic (hand-rolled), [Pico CSS](https://picocss.com/), [Tailwind](https://tailwindcss.com/), [DaisyUI](https://daisyui.com/)
- **Template engines** — [ErlyDTL](https://github.com/erlydtl/erlydtl) (Django-style) or [bbmustache](https://github.com/soranoba/bbmustache)
- **Database backends** — SQLite (default) or PostgreSQL stub ready to activate
- **Model generator** — `belongs_to`, `has_many`, and `many_to_many` relationships with reverse lookups and AND-logic tag search
- **Hot reload** — file watcher auto-recompiles `.erl` and `.html` files in the running VM with no restart
- **Debian 12 installer** — `cowboy-up setup` installs Erlang 26 from the RabbitMQ repo, rebar3, sqlite3, and inotify-tools
- **75 tests** — field parser, config normalisation, model generation, and template coverage all tested

---

## Requirements

- Python 3.8+
- [pipx](https://pipx.pypa.io/) (recommended) or pip

The generated projects require Erlang 26 and rebar3. Run `cowboy-up setup` on a fresh Debian 12 machine to install everything.

## Quick start

```bash
# Scaffold a new project interactively
cowboy-up new "my app"

# Or specify everything up front
cowboy-up new "bible site" --css pico --templates erlydtl --db sqlite

# Start the dev server (compiles, starts Cowboy, watches for changes)
cd bible_site
rebar3 as dev shell
```

Open http://localhost:8080. Edit any `.erl` or `.html` file and the change is live in ~150ms without restarting.

## Commands

### `cowboy-up new`

```
cowboy-up new "app name" [--css basic|pico|tailwind|daisyui]
                         [--templates erlydtl|mustache]
                         [--db sqlite|postgres]
                         [--no-prompt]
```

Scaffolds a complete project in a new directory named after the app. App name is normalised — `"Bible Site"` becomes `bible_site/`. When flags are omitted and stdin is a TTY, you are prompted to choose interactively.

### `cowboy-up model`

```
cowboy-up model <ModelName> <table> [field:type ...]
```

Generates `src/models/<name>.erl` with `all/0`, `find/1`, `create/1`, `update/2`, `delete/1`, and `to_map/1`, plus association functions for any relationships declared.

**Field types:**

| Spec | Column added? | What it generates |
|---|---|---|
| `title:text` | yes — TEXT | plain column |
| `count:integer` | yes — INTEGER | plain column |
| `price:real` | yes — REAL | plain column |
| `data:blob` | yes — BLOB | plain column |
| `book_id:belongs_to:Book` | yes — INTEGER FK | `book(Id)` — fetch parent Book |
| `teachings:has_many:Teaching` | no | `teachings(Id)` — fetch child records |
| `tag:many_to_many` | no — creates join table | `tag/2`, `tags/1`, `has/1` |

**Examples:**

```bash
# Parent model with reverse lookup
cowboy-up model Book books \
    title:text testament:text abbrev:text chapter_count:integer \
    teachings:has_many:Teaching

# Child model with FK, tags, and AND-logic search
cowboy-up model Teaching teachings \
    book_id:belongs_to:Book \
    title:text speaker:text filename:text duration_s:integer \
    tag:many_to_many
```

**`find/1` dual-clause pattern:**

```erlang
% By primary key
teaching:find(1).
% → {ok, #{id => 1, title => <<"Grace">>, ...}}

% By any column — returns a list
teaching:find({speaker, <<"John Smith">>}).
% → {ok, [#{...}, #{...}]}
```

**Tag search with AND logic:**

```erlang
% Teachings that have ALL of these tags
teaching:has([<<"sermon">>, <<"grace">>, <<"faith">>]).
% → {ok, [TeachingMap, ...]}

% Attach a tag (creates the tag if it doesn't exist)
teaching:tag(1, <<"sermon">>).

% List tags on a teaching
teaching:tags(1).
% → {ok, [<<"grace">>, <<"sermon">>]}
```

The `has/1` query uses relational division (`GROUP BY` + `HAVING COUNT(DISTINCT) = N`) so all listed tags must be present, not just any of them.

Also prints migration SQL to paste into `src/<app>_db.erl` and a shell usage summary.

### `cowboy-up model add-field`

```
cowboy-up model add-field <ModelName> <field:type>
```

Shows the exact manual edits needed to add a field to an existing model — does not modify files. Displays:

1. The migration tuple to add to `_db.erl`, with the `ALTER TABLE` statement and the live shell command to apply it without restarting
2. The exact `-define(FIELDS, ...)` line to find and what to change it to
3. The exact `to_map([...])` pattern to find and what to change it to
4. The map body key to insert
5. How to update `create/1` and `update/2` if the field needs to be writable
6. The `l(module).` shell command to hot-reload

```bash
cowboy-up model add-field Teaching notes:text
```

### `cowboy-up setup`

```
cowboy-up setup
```

Installs everything needed on Debian 12 / bookworm:

- System packages: `curl`, `build-essential`, `sqlite3`, `inotify-tools`
- Erlang 26 from the RabbitMQ apt repository (all required OTP applications)
- rebar3 (downloaded from S3, installed to `/usr/local/bin`)

Requires `sudo`. Prompts for confirmation before making any changes.

## Hot reload

The dev profile starts a file watcher that monitors `src/` and `priv/`:

| File type | Action |
|---|---|
| `.erl` | `compile:file/2` → `code:soft_purge` → `code:load_abs` |
| `.html` | ErlyDTL recompiles all templates (extends/includes are inlined at compile time) |
| `.css`, `.js`, `.json`, `.ts`, `.scss` | logged — hook SSE/WebSocket browser reload here |

Changes go live in ~150ms. Watcher only starts when the `fs` app is present (dev profile) — production releases exclude it automatically.

Start with:

```bash
rebar3 as dev shell
```

## Database

### Migrations

Migrations run automatically on startup and are idempotent — tracked in a `schema_migrations` table.

Add new migrations at the bottom of `migrations()` in `src/<app>_db.erl`:

```erlang
migrations() ->
    [
        {"20260101_001_create_books",
         "CREATE TABLE books (
            id         INTEGER PRIMARY KEY AUTOINCREMENT,
            title      TEXT NOT NULL,
            created_at TEXT DEFAULT (datetime('now'))
          );"},

        {"20260102_002_add_notes_to_books",
         "ALTER TABLE books ADD COLUMN notes TEXT;"}
    ].
```

### esqlite notes

esqlite 0.8.x returns rows as **lists**, not tuples. Always pattern match accordingly:

```erlang
%% Correct
Items = [#{id => Id, name => Name} || [Id, Name] <- Rows].

%% Wrong — will silently produce []
Items = [#{id => Id, name => Name} || {Id, Name} <- Rows].
```

### Switching to PostgreSQL

1. Uncomment `epgsql` and `poolboy` in `rebar.config`
2. Add them to the `applications` list in `.app.src`
3. Set `{db_backend, postgres}` in `config/sys.config`
4. Fill in the `postgres` config block in `sys.config`

The `<app>_db:q/1,2` and `<app>_db:exec/1,2` API is identical regardless of backend.

## Nginx

### Development

```bash
# Edit nginx/dev.conf — update the alias path to your priv/static dir
sudo ln -s $(pwd)/nginx/dev.conf /etc/nginx/sites-enabled/myapp
sudo nginx -s reload
```

### Production

```bash
# Obtain TLS certificate first
sudo certbot --nginx -d yourdomain.com

# Edit nginx/prod.conf — update domain and alias paths
sudo ln -s /path/to/myapp/nginx/prod.conf /etc/nginx/sites-enabled/myapp
sudo nginx -t && sudo nginx -s reload
```

Audio files under `/static/audio/` are served directly by Nginx with byte-range support so browser audio players can seek without buffering the full file.

## Deployment

```bash
rebar3 release

# Edit scripts/<app>.service — update WorkingDirectory and ExecStart paths
sudo cp scripts/<app>.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable --now <app>

# Logs
sudo journalctl -u <app> -f
```

Templates live in `cowboy_up/templates/`. Adding a new CSS theme means creating a directory under `templates/css/<theme>/` with `base.html.tmpl`, `home.html.tmpl`, `about.html.tmpl`, `error.html.tmpl`, and `app.css.tmpl`, then adding the theme name to `CSS_CHOICES` in `config.py`.

Adding a new database backend means creating `templates/db/<backend>/db.erl.tmpl` and adding the backend to `DB_CHOICES` in `config.py`.
